# option-side-effect-analysis

## 何をするか

対応方針の副作用を網羅列挙する。変更点が他の処理にどう波及するかを体系的に確認する。

## 実行手順

1. approach-plan.md の採用方針を確認する
2. 変更対象を明確にリストアップする（フィールド / メソッド / クラス / オブジェクト）
3. 以下の副作用カテゴリを順番に確認する:

   **Validation Rule**:
   - 変更フィールドが入力規則の formula に含まれていないか
   - 変更後に入力規則が意図せず発動する/しなくなる可能性

   **Approval Process / Escalation**:
   - 変更フィールドが承認プロセスの条件・アクションに関与していないか

   **Assignment Rule**:
   - 変更フィールドが割り当てルールの条件に含まれていないか

   **Trigger / Flow の連鎖**:
   - 変更後に別の Trigger / Flow が意図せず起動しないか
   - DML が重複・競合しないか

   **通知・連携**:
   - メール送信・Chatter 通知・Platform Event 送信が意図せず発火しないか
   - 外部連携（Callout）が副作用として動かないか

   **UI 表示**:
   - LWC / Aura の条件表示・非表示が変わらないか
   - ページレイアウトの表示条件への影響

4. 副作用ありと判定したものは対応方針に反映する（抑制 / 許容 / 意図的に発生させる）

## 出力

approach-plan.md「副作用分析」セクションに追記:

| 副作用カテゴリ | 対象 | 影響評価 | 対応方針 |
|---|---|---|---|
| Validation Rule | {フィールド名} | あり / なし | {対応内容 / 無視} |
| Trigger/Flow | ... | ... | ... |
