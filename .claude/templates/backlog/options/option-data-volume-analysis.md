# option-data-volume-analysis

## 何をするか

変更対象オブジェクトのデータ件数・大量データ時の影響を評価する。

## 実行手順

1. 変更対象オブジェクトのデータ件数を Sandbox で確認する:
   ```bash
   sf data query --query "SELECT COUNT() FROM {ObjectName__c}" --target-org {sandbox-alias} --json
   ```
2. 件数に応じた影響評価を行う:
   - **〜1000件**: 通常処理で問題なし
   - **1万件〜**: SOQL の WHERE 条件・インデックスの有無を確認
   - **10万件〜**: バルク処理・バッチ対応の必要性を検討
   - **100万件〜**: 大量データによるガバナ制限・タイムアウトリスクを精査
3. 今回の変更（SOQL 追加・フィールド追加・処理追加）が大量データ環境で問題を起こすか評価する
4. **本番 SELECT 参照は必須（「検討」ではない）**: Sandbox のレコードはテストデータであり本番の実件数・入力傾向を表さない。上記 1〜3 の Sandbox 件数は「処理負荷の試算（性能観点）」にのみ使ってよく、**「実際どのくらい入力・使用されているか」等の業務実態の根拠として提示してはならない**。業務実態を主張する場合は必ず本番 SELECT 参照（option-prod-select-reference）で件数・分布を確認する。実施できない場合は `[要確認: 本番データ未確認]` を付け、Sandbox 件数だけで「〜件しかない」「ほぼ使われていない」等の結論を断定しない。

## 出力

investigation.md に追記:

## データ量分析

- {オブジェクト名}: Sandbox での件数 {N} 件（**処理負荷試算のみに使用。業務実態の根拠にしない**）
- 本番件数: {本番 SELECT 参照で確認した値} / `[要確認: 本番データ未確認]`
- 影響評価: 問題なし / 要注意（{理由}） / 要バッチ対応
- 推奨対応: {なし / SOQL 最適化 / バルク対応 / バッチ処理}
