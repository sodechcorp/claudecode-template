# option-customer-questions

## 何をするか

Phase A の `### 業務要件の確認事項`（Q1./Q2.）を業務語彙に変換した対外向「顧客確認事項ドラフト」を生成する。**ユーザーが明示的に依頼した場合のみ生成し、デフォルトの Phase A 挙動は一切変えない。**

## Claude の責務範囲

**Claude はドラフトを提示するのみで、顧客への送信・Backlog への投稿は行わない。** 生成物の見出しには必ず「ドラフト・未送信・要人間レビュー」を明示する。送信可否・文面の最終判断は人間が行う。Backlog 書き込み系・メール送信系 MCP は使わない（社内向け・社外向けを問わず。PreToolUse hook で技術的にハードブロック済み）。

## 実行手順

1. approach-plan.md の `### 業務要件の確認事項`（Q1./Q2.）を読む
   - Q が 0 件、または「なし」の場合は生成せず「確認事項なしのためドラフト対象なし」とだけ提示して終了する
2. 各 Q を対外向文面へ変換する:
   - オブジェクト API 名（`__c` 等）・Apex/LWC/クラス名・技術用語・案A/案B のような設計都合の言い回しを禁止し、業務ユーザーが答えられる業務的な問いに書き換える
   - ラベル（業務呼称）を使い、API 名は使わない
   - 各項目は「背景1行＋問い1行」の粒度でまとめる
3. 見出しに「ドラフト・未送信・要人間レビュー」を必須明示してチャットに提示する
4. `docs/logs/{issueID}/customer-questions-draft.md` に保存する

## 出力

`docs/logs/{issueID}/customer-questions-draft.md` を新規作成（approach-plan.md と同じ課題ログフォルダ）:

```
# 顧客確認事項ドラフト（未送信・要人間レビュー）

> このドラフトはまだ顧客に送信されていません。内容を確認のうえ、人間が送信可否を判断してください。

- 背景: {業務語彙での背景1行}
  問い: {業務語彙での問い1行}
- 背景: ...
  問い: ...

---
（内部 Q 番号との対応: Q1 → 上記1件目 / Q2 → 上記2件目 ...）
```
