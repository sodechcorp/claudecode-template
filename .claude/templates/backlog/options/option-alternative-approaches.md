# option-alternative-approaches

## 何をするか

推奨案 A に加えて対応方針 B / C を生成し、比較検討できる状態にする。

## 実行手順

1. 現在の推奨案 A を確認する（approach-plan.md の推奨案）
2. 以下の軸から別案を考える:
   - **解決レイヤー**: LWC 側 / Apex 側 / Flow 側 / 設定変更のみ
   - **変更範囲**: 最小限修正 / 整合性のある拡張 / 根本対応
   - **既存パターン**: 踏襲 / 独自設計
3. 案 B・C を生成する（各案: 概要・メリット・デメリット・見込み工数）:

   > **見込み工数は `sf-effort-estimator` に各案の対応方針サマリを渡して Agent 委譲で算出する（planner が作業を分解して積み上げない）。** 渡すパラメータ: `対応方針`=各案概要 / `スコープ`=変更ファイル数・流用可否 / その他は backlog-planner.md §A-2.5 参照。

   **バグの場合**:
   - 案 B: 根本原因に踏み込む修正（リファクタリングを伴う・影響範囲が広い）
   - 案 C: 設定変更・データ補正で回避（コード変更なし。可能な場合のみ）

   **追加要望の場合**:
   - 案 B: 独自設計（既存パターンとの差分とその根拠を明記）
   - 案 C: 別のアーキテクチャ（Flow 化 / Platform Event 化 / batch 化 等）

4. 各案について推奨案 A との比較表を作成する
5. approach-plan.md の「対応方針」セクションを更新する（案 B・C を追記）

## 出力

approach-plan.md の「対応方針」セクションを更新（案 B・C を追記）:

> **見込み工数列**は各案ごとに `sf-effort-estimator` を Agent 委譲して算出した単一値（例: `1.5h`）を入れる。「実装 Xh + テスト Yh + …」形式の作業分解表は禁止。

| リスク | ... | ... | ... |
