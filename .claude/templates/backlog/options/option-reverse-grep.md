# option-reverse-grep

## 何をするか

逆参照の**アンカー対象を2モードで切り替える**:
- **通常モード**: 変更対象の API 名・メソッド名・フィールド名を `force-app/` 全体から Grep で逆参照する。
- **root-cause-anchor モード**（backlog-investigator Step C-2 から指定された場合）: 「変更対象名」ではなく**根本原因の共有コンポーネント名（共有メソッド・共通ユーティリティ）**をアンカーにして全呼び出し元を洗い出す。目的は「兄弟入口（報告症状が同一条件で再現する別入口）」の特定。

## 実行手順

1. アンカー名リストを作成する:
   - 通常モード: 変更対象の名前（API 名・メソッド名・フィールド名）
   - root-cause-anchor モード: 根本原因の共有コンポーネント名（クラス.メソッド・共有メソッド名）
2. 各名前について `force-app/` 全体を Grep で検索する:
   - Apex (`*.cls`, `*.trigger`)
   - LWC (`*.js`, `*.html`)
   - Aura (`*.cmp`, `*.js`)
   - VisualForce (`*.page`)
   - Flow (`*.flow-meta.xml`)
   - 入力規則 (`*.validationRule-meta.xml`)
   - 承認プロセス (`*.approvalProcess-meta.xml`)
   - 割り当てルール (`*.assignmentRules-meta.xml`)
3. ヒットした各箇所（＝呼び出し元）について判定する:
   - **root-cause-anchor モードの場合**: 各呼び出し元が「課題で報告された症状そのものを再現するか」で振り分ける:
     - 症状が再現する（同一根本原因を共有）→ **(A) 兄弟入口** → investigation.md「影響範囲」に区分 S で追記（同一症状が再現する根拠を `ファイル名:行` で明記）
     - 症状が再現しない／別症状／無関係 → **(B)** → 影響なしとしてスキップ理由を記録、または派生事項として記録
   - **通常モードの場合**: 影響あり → investigation.md「影響範囲」に追記（区分 I） / 影響なし → スキップ理由を簡潔に記録

## 出力

investigation.md「影響範囲」セクションに追記:

| ヒットファイル | 行 | 区分（S=兄弟入口/I=通常影響/なし） | 影響判定 | 対応要否 |
|---|---|---|---|---|
| ... | ... | ... | ... | ... |
