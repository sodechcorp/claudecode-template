# option-knowledge-extraction

## 何をするか

今回の課題対応で得た知見・却下案・指摘パターンを `docs/decisions.md` に残す。次の担当者が同種の問題に直面した時に活用できる知識を蓄積する。

## 実行手順

1. `docs/logs/{issueID}/discussion-log.md` を Read して今 Phase の議論・指摘・却下案を確認する（存在する場合のみ）

2. `docs/logs/{issueID}/approach-plan.md` の改版履歴テーブルと「採用方針」欄を Read する

3. 以下を「次の人に役立つ知見」として列挙する:

   **設計・実装の知見**:
   - 今回判明した制約（「この処理は〇〇の理由でこうしないと動かない」）
   - Salesforce の仕様で注意が必要な挙動
   - 回避策・ベストプラクティス
   - 想定と異なった動作・調査で判明した意外な事実

   **却下された案・ユーザーが好まないパターン**:
   - discussion-log.md で却下になった案と却下理由
   - ユーザーが明示的に「違う」「そうじゃない」と言った実装パターン
   - 「一見良さそうだが NGだった」設計上の落とし穴

   **繰り返し出た指摘・確認事項**:
   - 複数 Phase で同じテーマの指摘が出た場合はそのパターン
   - 「毎回業務側に確認が必要な判断ポイント」として汎用化できるもの

   → **「1 回限りの判断」や「この課題固有の事情」は記録不要**（decisions.md が肥大化するため）

4. `docs/decisions.md` に追記する（最上部に降順で追加）:

   ```markdown
   ## {YYYY-MM-DD}: {判断テーマ} ({issueID})

   **判断**: {採用した方針・実装パターン}
   **却下案**: {検討したが却下した案と却下理由}（なければ省略）
   **ユーザー指摘**: {対応中に出たユーザー指摘で再利用価値があるもの}（なければ省略）
   **再発防止・注意点**: {将来同種課題が来たら参照すべきパターン・注意事項}
   ```

5. `docs/decisions.md` が存在しない場合は新規作成する

6. 記録したことを test-report.md（Phase 5 実行時）または最終成果物に記録する

## 出力

`docs/decisions.md` に追記 + 最終成果物に追記:

## 知見抽出・decisions.md 更新

- 記録件数: {N} 件
- 主な内容: {追記したエントリの概要}
