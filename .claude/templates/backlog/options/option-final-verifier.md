# option-final-verifier

## 何をするか

課題本文の期待結果と実挙動だけを比較して「この課題は解決されたか」を blind 判定する。実装経緯を知らない第三者視点での最終確認。subagent として実行することで先入観を排除する。

## 実行手順（subagent 化必須）

**このオプションは必ず `backlog-blind-final-verifier` subagent を Task ツールで起動して実行する。parent 内で直接実行してはならない（blind 性が崩れる）。**

subagent への引き渡し情報:
1. 課題 ID（Backlog issue key）
2. 課題本文の全文
3. 全コメントのテキスト（後付け要件を含む全内容）
4. After エビデンス（動作確認結果をテキストで記述するか、読み取り可能なファイルのパスを指定。スクリーンショット等のバイナリはパス指定可だが、verifier 側で内容は判定材料にならない）
5. 以下を明示する: 「実装の経緯・実装計画は一切伝えない。あなたは課題本文と After エビデンスだけで解決しているかを判定してください」
6. 実施日時（現在日時: YYYY-MM-DD HH:MM JST を記入。verifier は tools: Read のみのため自力取得不可）

subagent が判定する内容:
- 課題本文の期待結果が満たされているか（Yes / No / 部分的）
- コメントに後付けで追加された要件が全て対応されているか
- 「見た目は直っているが業務的に解決されていない」点があれば指摘

## 出力

test-report.md に追記:

## blind 最終解決判定

- 実施日時: {YYYY-MM-DD HH:MM JST}
- 判定: 解決済み / 未解決（{理由}）/ 部分的解決（{未対応: ...}）/ 判定不能（エビデンス未提供）
- 見落とし指摘: なし / あり（{内容}）

判定が「解決済み」でない場合: 対応内容を implementation-plan.md に追記して再実装する。
判定が「判定不能: エビデンス未提供」の場合: After エビデンスを取得してから再実行する。
