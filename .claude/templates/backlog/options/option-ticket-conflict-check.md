# option-ticket-conflict-check

## 何をするか

本番リリース準備の対象課題（`{issueID}`）が変更するメタデータ・コンポーネントが、**他の進行中課題（未リリース）が触っているものと衝突していないか**を Backlog read-only MCP で確認する。同じ組織に対して複数課題が並行対応されている場合、片方のリリースがもう片方の未リリース変更を巻き込む／上書きするリスクを事前に検知する。

**このオプションは read-only**。Backlog への書き込み（コメント投稿・ステータス変更）は行わない（`pre-operation.js` がハードブロック対象でもある）。

## 実行手順

1. **自課題の変更対象を確定する**（release-preparer Phase 1 で確定済みの資材マニフェストを使う）:
   - 変更ファイルパス一覧（`force-app/**`）
   - 変更対象の API 名（Apex クラス名・Flow API 名・オブジェクト名・フィールド API 名等。ファイルパスから逆算）

2. **進行中の他課題を取得する**（Backlog read-only MCP）:
   ```
   mcp__backlog__get_issues（プロジェクトID指定・statusId で「処理中」「未対応」等の未完了ステータスに絞り込み・自課題 {issueID} は除外）
   ```
   - 件数が多い場合は直近更新順（`sort: updated`, `order: desc`）で上位 N 件（目安 30 件）に絞る。全件走査が必要な規模の場合はユーザーに絞り込み条件（担当者・カテゴリ等）を確認する

3. **各候補課題の関連コミット・PR を確認する**（分かる範囲で）:
   - `mcp__backlog__get_git_repositories` / PR がある場合は `mcp__backlog__get_pull_requests` で該当課題番号を含む PR タイトル・ブランチ名を検索
   - 課題本文・コメントに記載されたファイルパス・コンポーネント名を `mcp__backlog__get_issue` / `get_issue_comments` で確認する（本文からの手がかりに限る。Backlog に紐づく Git リポジトリの実ファイル差分までは追わない＝重すぎるため対象外）

4. **突き合わせる**:
   - 自課題の変更対象ファイル/API名と、候補課題の言及ファイル/API名が一致 or 包含関係にあれば「競合候補」としてマークする
   - 一致の確度が低い（キーワードの部分一致のみ等）場合は「要目視確認」として区別する

5. **重大度を判定する**:
   | 判定 | 条件 |
   |---|---|
   | 高 | 同一ファイル・同一メソッド/フィールドを両課題が変更（本文明記あり） |
   | 中 | 同一ファイル・別箇所を変更（本文から推定） |
   | 低 | 同一オブジェクト・別コンポーネントで関連性が薄い |
   | 情報不足 | 候補課題側の記載が薄く判定できない |

## 出力

`docs/logs/{issueID}/release-plan.md`「## チケット競合チェック」セクションに追記:

```markdown
## チケット競合チェック

確認対象: 進行中課題 {N} 件（{確認日時}時点）

| 競合候補課題 | 重大度 | 競合対象 | 根拠 |
|---|---|---|---|
| {課題ID} | 高/中/低/情報不足 | {ファイル/API名} | {課題本文・コメントの該当箇所引用} |

判定: 競合なし / 競合あり（要人間確認）
```

**競合あり（高・中）と判定した場合**: release-plan.md 生成を中断せず記録した上で、完了報告時にユーザーへ明示的に警告する（「{課題ID} と競合の可能性があります。リリース順序を確認してください」）。デプロイ CLI は生成するが、手順書冒頭に警告を明記する。
