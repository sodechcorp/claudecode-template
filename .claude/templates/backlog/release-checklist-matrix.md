# 本番リリース チェックリスト・マトリクス（前 → 実行 → 後）

`release-preparer` Phase 5 が参照する、**リリース前 → リリース実行 → リリース後**の順に整理したチェック項目定義。資材種別（メタデータタイプ）ごとに検証方法・注意点が異なるため、Phase 1 で確定した資材マニフェストに**実際に含まれる種別のみ**を release-plan.md に転記する（含まれない種別の行は書かない）。

> **役割分担**: 権限付与の詳細は [new-metadata-permissions-checklist.md](../common/new-metadata-permissions-checklist.md)、デプロイ順序は [option-deployment-dependency-check.md](options/option-deployment-dependency-check.md)、テスト観点は [test-pattern-map.md](test-pattern-map.md) が正本。本ファイルはそれらを**リリース文脈で束ねる**ものであり、権限・順序・テスト観点の詳細ロジックを再定義しない（参照する）。
>
> **read-only 原則**: release-preparer は本番へ read-only 操作しか行わない。リリース後チェックのうち「本番で実行する検証」は**手順書に記載して人間が実行**する。エージェントが本番でテスト実行・DML・デプロイをすることはない。

---

## A. リリース前チェック（pre-release・全資材共通）

デプロイ前に人間が確認する。release-preparer は各項目の**現時点の状態**を read-only で確認して埋められるものは埋め、埋められないものはチェックボックスのまま残す。

- [ ] Sandbox でのテスト完了（`test-report.md` の総合判定が PASS）
- [ ] `/test` の詳細証跡取得済み（`evidence/` に before/after が揃っている）
- [ ] デプロイ対象資材の確定（Phase 1 資材マニフェストと git diff が一致）
- [ ] デプロイ順序の確認（Phase 1 の依存関係判定。分割要ならその順序）
- [ ] 影響範囲の確認（Phase 2 の各 option 結果にリリースを止める要素がない）
- [ ] チケット競合チェック: 問題なし（Phase 3。競合ありなら承知の上か）
- [ ] 本番環境ドリフト確認: 問題なし（Phase 4。ドリフトありなら承知の上か）
- [ ] ロールバック手順の確認（`option-rollback-readiness` 準拠・コミットハッシュ記録の準備）
- [ ] お客様確認サイン取得済み（バグ・仕様変更を伴う場合）
- [ ] リリース日時・実施者・立ち会い者の確定
- [ ] リリース時間帯の業務影響確認（営業時間中か・バッチ稼働時間と重ならないか）

## B. リリース実行手順（execution・全資材共通）

**この手順は人間が実行する。release-preparer は実行しない。**

1. **直前記録**: `git log -1 --pretty=format:'%H'` でロールバック用コミットハッシュを記録する
2. **dry-run（必須）**: `sf project deploy start --dry-run --source-dir force-app --target-org <本番エイリアス> --test-level RunLocalTests` で 0 errors を確認
3. **デプロイ実行**: dry-run 成功後に `--dry-run` を外して実行
4. **デプロイ順序**: 一括不可の場合は Phase 1 の依存順序（構造 → ロジック → UI → 自動化 → 権限 → レイアウト）で分割実行
5. **結果確認**: `sf project deploy report --target-org <本番エイリアス>` で成功を確認
6. **管理画面手動操作**: ソースデプロイ対象外の資材（Phase 1 で分離したもの）があれば、手順書の管理画面操作セクションに従って実施

## C. リリース後チェック（post-release・全資材共通）

デプロイ完了直後に人間が本番で確認する。

- [ ] デプロイ成功確認（`sf project deploy report` の結果を記録）
- [ ] 資材種別別のリリース後検証（下記 §D の該当種別を実施）
- [ ] リリース後エビデンス取得（本番での動作確認結果を `evidence/release-verification/` 相当に保存）
- [ ] エラーログ確認（デプロイ直後の Setup → デバッグログ・例外メールに異常がないか）
- [ ] 関係者への完了連絡・リリースノート展開
- [ ] 問題発生時のロールバック判断基準の再確認（誰がどの症状で判断・実行するか）

## D. 資材種別別チェック（該当種別のみ転記）

Phase 1 資材マニフェストに含まれる種別だけを release-plan.md に転記する。「リリース後検証方法」は本番で人間が実施する。

### Apex クラス
- **リリース前**: dry-run で 0 errors・ガバナ制限/FLS/CRUD 確認済み・テストクラス整備とカバレッジ（推奨 75%以上）
- **リリース後検証**: 対象テストクラスを本番で再実行（`sf apex run test --class-names {テストクラス} --target-org <本番>`）。呼び出し元機能を UI/SOQL で実挙動確認
- **注意点**: `classAccesses` の権限付与（システム管理者必須・他プロファイルはユーザー確認）。バッチ/スケジューラの内部実行は直接アクセス不要

### Apex トリガー
- **リリース前**: dry-run・before/after 各経路・バルク 200 件・再帰防止の確認
- **リリース後検証**: トリガー対象オブジェクトへ本番で代表操作を1件行い、期待どおり動くか + 既存レコード一括操作で異常がないか（少量で）
- **注意点**: DML 競合・無限ループ。他トリガー/Flow との連鎖実行に注意

### Flow（画面フロー / レコードトリガー / スケジュール）
- **リリース前**: dry-run・起動条件/参照先/API名確認・分岐各経路のテスト済み
- **リリース後検証**: 本番で起動条件を満たす操作を行い、レコード作成/更新結果を SOQL で確認。画面フローは実画面で操作確認
- **注意点**: **Active/Latest バージョン問題**（Metadata API はデプロイ後に有効バージョンが切り替わる。デプロイ後に本番で「有効」バージョンが意図どおりか必ず確認）。スケジュール実時刻起動は即時検証困難＝別途フォロー

### LWC / Aura
- **リリース前**: dry-run・`with sharing` 確認・埋め込み先（レイアウト/フレックスページ）の確認
- **リリース後検証**: 本番の実画面で表示・操作・エラー表示を確認（スクショ保存）。コントローラ挙動は SOQL/操作結果で確認
- **注意点**: 埋め込み先ページ/コンポーネントへのアクセス権限。キャッシュで旧版が見える場合はハードリロード

### Visualforce ページ
- **リリース前**: dry-run
- **リリース後検証**: 本番で画面表示・入力処理を確認
- **注意点**: `pageAccesses` の権限付与。VF の FLS 自動適用有無を確認

### カスタムオブジェクト
- **リリース前**: dry-run・CRUD/FLS/共有ルールの確認・デフォルトレイアウト/タブ/アプリ追加の確認
- **リリース後検証**: 本番で対象オブジェクトの一覧表示・レコード作成が可能か。SOQL で件数確認
- **注意点**: `objectPermissions`（CRUD）の付与。タブ・ページレイアウト・Lightning App への追加

### カスタム項目
- **リリース前**: FLS 設定（readable/editable）・ページレイアウト配置・コンパクト/検索レイアウト確認
- **リリース後検証**: 本番の対象画面で項目が表示/編集できるか（対象プロファイルで確認）。値が保存され SOQL で取得できるか
- **注意点**: `fieldPermissions`（FLS）。項目がレイアウト未配置だと「作ったのに見えない」が起きる。複数レイアウトの表示要否

### レコードタイプ
- **リリース前**: dry-run・`recordTypeVisibilities`（プロファイル割当）・レイアウト割当・ピックリスト値制限の確認
- **リリース後検証**: 本番でレコード作成時にレコードタイプが選択でき、正しいレイアウト/ピックリスト値になるか
- **注意点**: プロファイルごとの visible/default。ページレイアウト割当との整合

### 権限セット / プロファイル
- **リリース前**: デプロイ対象（メタデータ）に含まれるか確認。差分（付与/剥奪される権限）の把握
- **リリース後検証**: 対象ユーザで Login As し、本番で意図どおりの表示/操作権限になっているか差分確認
- **注意点**: **権限セットは自動作成/更新しない（ユーザー明示指定時のみ）**。プロファイル変更は全ユーザーへの影響が広いため差分を厳密に確認

### ページレイアウト
- **リリース前**: レイアウト割当（プロファイル別・レコードタイプ別）の確認
- **リリース後検証**: 本番で対象プロファイル/レコードタイプの画面配置を目視確認
- **注意点**: 複数レコードタイプ・プロファイル別割当。管理画面設定のみで完結する場合はデプロイ不要（手順書の管理画面操作へ）

### 入力規則（Validation Rule）
- **リリース前**: dry-run・違反データで弾く/正常データが通過するテスト済み・境界値
- **リリース後検証**: 本番で違反データ入力が弾かれ・正常データが通ることを1件ずつ確認。エラー文言の表示確認
- **注意点**: 副作用（他項目変更で意図せず発動）。既存レコード編集時の遡及発動

### カスタム設定 / カスタムメタデータ型
- **リリース前**: dry-run・階層型/リスト型の違いを把握・参照側への反映確認
- **リリース後検証**: 本番で値を SOQL/画面で確認。利用側（Apex/Flow）が正しく参照するか
- **注意点**: カスタム設定の**データ値はメタデータデプロイに含まれない**場合がある（別途投入要）。カスタムメタデータ型はレコードもデプロイ対象か確認

### レポート / ダッシュボード
- **リリース前**: フォルダ・可視性・参照データの確認
- **リリース後検証**: 本番で表示・データ更新・キャッシュ最新化を確認
- **注意点**: フォルダアクセス権限。外部データ/リモート接続がある場合の疎通

---

## E. マトリクスにない種別が出た場合

上記に該当種別がない資材（例: メールテンプレート・承認プロセス・共有ルール・Static Resource 等）は、`§A/§B/§C` の共通チェックを適用した上で、「リリース後検証方法」を release-preparer が **[要確認]** マーカー付きで起案し、人間に検証方法の確認を促す（推測で断定しない）。
