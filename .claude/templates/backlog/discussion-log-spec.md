# discussion-log.md 仕様

各エージェントが Phase 末尾確認プロトコルを実行した直後に、`docs/logs/{issueID}/discussion-log.md` へ追記するための共通仕様。

**目的**: 「最初こう→指摘されてB案→ユーザー再指摘でC案」の経緯を時系列で残し、同じ調査・同じ指摘・同じ却下済み方針を次 Phase / 次回対応で繰り返さない。

---

## ファイル位置

```
docs/logs/{issueID}/discussion-log.md
```

1 課題 1 ファイル。最初に追記するエージェントが存在しない場合は新規作成する。

---

## 記録フォーマット

```markdown
# {issueID} 議論ログ

## Phase {N}: {Phase 名}

- {YYYY-MM-DD HH:MM} [{発信者} → {受信者}] {種別}: 「{内容}」{→ 対応}
- ...
```

### 発信者・受信者の表記

| パターン | 表記例 |
|---|---|
| エージェントからユーザーへの質問 | `[investigator → user]` |
| ユーザーからエージェントへの回答 | `[user → planner]` |
| エージェント内部の発見・変更 | `[implementer]` |

### 種別タグ

| 種別 | 使うとき |
|---|---|
| `Q起票` | エージェントが業務要件 Q を起票した |
| `回答` | ユーザーが Q に回答した（生引用に近い形で） |
| `指摘` | ユーザーが方針・内容に異議・補足を出した（生引用に近い形で） |
| `却下` | ユーザーまたはエージェントが案を却下した |
| `発見` | 調査・実装中に判明した重要な新事実 |
| `案提示` | エージェントが方針案を提示した（1〜2 行要約） |
| `変更` | 方針・判断を変更した（理由を必ず書く） |
| `落とし穴` | 「これをするとこうなる」という注意点・地雷を発見した（pitfalls.md 自動還流の対象） |
| `ハマり` | 意図した動作にならず原因特定に時間がかかった経緯（pitfalls.md 自動還流の対象） |

## 記録ルール

### 何を残すか（必須）

- **ユーザー指摘・却下**: 「違う」「やり直し」「B案にして」等の内容を **生引用に近い形** で残す
  - 例: `[user → planner] 指摘: 「既存レコードへの影響を考慮していない。過去データはどう扱う？」`
- **却下された案**: 「案Aを却下した理由」を 1 行で残す
  - 例: `[user → planner] 却下: 「案A（バッチ処理）→ リアルタイム反映が要件のため不可」→ 案B へ`
- **重要な発見**: 調査中に判明した仕様・制約・意外な挙動で、後 Phase にも影響するもの
  - 例: `[investigator] 発見: 「Account.CustomField__c は入力規則で空白禁止。デフォルト値の設定が必須」`
- **方針変更の経緯**: Phase をまたいで方針が変わった理由
  - 例: `[implementer] 変更: 「トリガー実装」→「Flow 実装」 理由: 「既存トリガーとの競合リスク回避（investigator Phase 1 発見参照）」`

### 何を残さないか

- Q番号テーブルの内容（`approach-plan.md` 側に記録されているため重複不要）
- 問題なく承認されたフロー（「OK です」→ 次 Phase、だけのやり取り）
- 単純な実装詳細（命名・コードの書き方レベル）

### 書くタイミングと責任者分担

| イベント / 種別タグ | 発生タイミング | 追記責任者 | 追記タイミング |
|---|---|---|---|
| `Q起票` / `案提示` / `発見` / `変更` / `落とし穴` / `ハマり` | エージェント実行中 | 当該 Phase の agent | Phase 末尾確認プロトコル直後に当該 Phase 分を一括追記 |
| `回答` / `指摘` / `却下`（ユーザー発話） | Phase 間のユーザー応答 | メインスレッド | ユーザー発話を受けた直後に即追記 |
| `指摘` / `却下`（Phase 内でユーザーから出た場合） | Phase 末尾確認プロトコルでのユーザー回答 | 当該 Phase の agent | Phase 末尾でユーザー回答を取得した直後 |

**原則:** agent 内部で発生したイベントは agent が記録、Phase 間のユーザー発話はメインスレッドが記録する。Phase 末尾確認プロトコルは agent 側のイベントとみなし agent が追記する（ユーザー回答も含めて当該 Phase の agent が拾う）。

### 読み込みタイミング（各エージェント Step 0a）

各エージェントは Step 0a の末尾で discussion-log.md の末尾 20 件（またはファイル全体が短ければ全件）を Read し、まだ成果物に反映されていない指摘がないかを確認してから作業を開始すること。

### 粒度

- ユーザー発話: 生引用に近い形（要約は最小限）
- エージェントの案・発見: 1〜2 行要約
- 長文の指摘は 3〜5 行に圧縮して本質を残す

---

## 追記の実行方法

```
docs/logs/{issueID}/discussion-log.md が存在しない場合:
  Write で以下のヘッダーから新規作成する:
  ---
  # {issueID} 議論ログ
  ---

存在する場合:
  末尾に追記（Edit ツールで既存行を変更しない）
```

---

## 読み込みタイミング（Phase 0d）

新規 Phase 開始時（Phase 0d）に、同 issueID フォルダ内の既存ファイルとともに必読する。
読み込み手順は `backlog.md §Phase 0d` を参照。
