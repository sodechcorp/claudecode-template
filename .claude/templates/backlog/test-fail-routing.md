# Phase 5 NG 時の戻り先テーブル

| NG の原因 | 戻り先 |
|---|---|
| 実装ミス・コードのバグ | Phase 4（実装修正） |
| 実装方針レベルの判断ミス | Phase 3（実装方針の再確認） |
| 対応方針レベルの設計ミス | Phase 2（対応方針の見直し） |
| テスト仕様・期待値・シナリオ自体の誤り（製品コードは正しい） | `test-spec.md` の期待値・シナリオ・前提を修正して `/test` を再実行（**Phase を戻さない**）。製品コードが正しいと確認でき、期待値修正のみで NG が解消すると見込める場合に限る |
| 判断できない・複合原因 | ユーザに戻り先をテキストで確認（「Phase 4 / Phase 3 / Phase 2 / 中止 のどれに戻りますか？」と質問） |

> **テスト仕様・期待値の修正は NG 理由が明確な場合のみ行う**（実装バグを隠すために `test-spec.md` の期待結果を書き換えてはならない。`/test` の設計原則と同じ）。判断に迷う場合は「判断できない・複合原因」行に従いユーザー確認する。

戻って修正した後は **NG TC ＋ 影響範囲の TC を再テスト**してから Phase 6 に進む。前回 OK かつ影響範囲外の TC は判定結果・証跡を流用（`--prev` オプション）してよい。判断が困難・影響範囲が広い場合は `--full` で全件再テストする。

> 修正後は必ず **Phase 6 で Sandbox に再デプロイ**してから `/test` を再実行すること（`/test` はデプロイしない）。

## ループ上限

Phase 5 への差し戻しは **最大 3 回まで**。4 回目に達した場合は「繰り返し NG が続いています。業務担当者との打ち合わせを推奨します」とユーザーに提案し、継続・中止の判断を求める。

**ループカウントのセッション跨ぎ保持**: `docs/logs/{issueID}/` 配下の退避済み `judgment-result.R{N}.json` の本数を通算カウントとする（ファイル数 = これまでの NG 修正回数）。セッションを跨いだ再開でもカウントはリセットしない。ループ上限に達した場合はユーザー確認を取る（test-report.md に「ループ上限到達」を記録してから確認）。

なお Phase 1.6 および Phase 3.5 のループ上限も同様に discussion-log.md の改版履歴から通算カウントする（各エージェントの定義を参照）。
