# お客様確認サインの種別別ルール

## Claude の責務範囲

**お客様への確認依頼・Backlog コメント返信・メールでのサイン取得は人間が実施する**。Claude は以下のみ行う:

- Phase 5 の再現テスト結果エビデンス（Before/After 対）を `test-report.md` に成果物として残す
- ユーザーに「お客様確認サインを取得してください（取得手段は Backlog コメント返信 / メール等、手段はユーザー判断）」とリマインドする
- ユーザーから「サイン取得済み」「サイン不要（種別が追加要望/その他）」のテキスト報告を受けた上で完了報告に進む

**Claude は Backlog コメントを投稿しない・お客様向け文面を勝手に起案して提示しない**。お客様向け文面の起案が必要な場合はユーザーから明示依頼があった時のみ、チャットでテキスト提示する（投稿はユーザーが手動）。

## 種別別の必要性

| `{issue_type}` | 取り扱い |
|---|---|
| バグ | **必須**。Claude は再現テスト結果（Phase 5 の 4.5-A エビデンス）を成果物に残し、ユーザーにサイン取得を依頼する。**ユーザーからサイン取得済みの報告を受けるまで完了報告しない** |
| 追加要望 | 推奨。Claude は UAT 実施予定の有無をユーザーに確認し、必要なら同様にリマインドのみ |
| その他 | 任意。Claude からのリマインドは省略可 |

## xlsx 更新（お客様確認）

ユーザーから「サイン取得済み」報告を受けた後、`{issue_type}` が `バグ` かつ `{xlsx_folder}` が未設定でなく `{issueID}` も変数名のまま展開されていない場合のみ実行（いずれかがリテラルのままならスキップ）:

```bash
python scripts/python/backlog-xlsx/update_records.py \
  --folder "{xlsx_folder}" --issue-id "{issueID}" \
  timeline --phase "お客様確認" \
  --source "顧客" \
  --content "確認サイン取得: {取得日・確認手段（ユーザーからの報告内容）}"
```

> ユーザーが取得手段（Backlog コメント / メール等）を明示しなかった場合は「お客様確認サイン取得（手段未指定）」と記録する。Claude から取得手段を推測しない。
