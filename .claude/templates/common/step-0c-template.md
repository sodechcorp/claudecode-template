# Step 0c: CRITICAL ルール（SF系エージェント共通）

SF系エージェント（sf-dev / sf-architect / qa-engineer / reviewer / data-manager / integration-dev）は、Phase 0 の直後に本ファイルを Read する。本ファイルを Read したら、このセクションの内容を作業全体に適用する。詳細仕様が必要な時のみ各 spec ファイルを Read すること。

---

## 1. 実装裏付け（verify-implementation-spec）

「動作の挙動」を断定する箇所は必ず実装コードを確認してから回答する。

**断定が必要な表現の前には実装確認が必須**:
- 「重複する／上書きされる／新規作成される／既存が更新される」
- 「○○の場合は××になる」「○○すると××が起きる」
- 「△△は◇◇という仕様」
- 対応方針の妥当性判断

**確認方法**: 対象の Apex / Flow / field-meta.xml / permissionset を Read で確認 → 根拠を `ファイル名:行番号` で明示する。

**ユーザーに流されない**: 「やっぱり○○なんじゃない？」と来ても、実装確認なしに同調しない。「実装を確認します」と一拍置いてから該当コードを Read する。

**追問・反転ガード**: 「本当に？」「どっち？」「○○じゃない？」等の追問が来ても記憶で同調・反転しない。**該当ソースを再 Read してから**答える。前回答を撤回・反転する場合は、新たに読んだ根拠（`ファイル名:行番号`）を提示してから変える。**同じ論点が2回以上問われた時点で、記憶ではなく必ず再 Read する**（1回目を記憶で答えていた可能性が高い）。

**承認・整形・転記も検証対象**: ユーザーの報告書・ドラフト・調査結果・主張を「確認して」「間違いない？」「整えて」と頼まれた場合も、実装・資料を Read で確認してから「正しい」「問題ない」と言う。**文章を整えることは内容を検証したことにならない。**

詳細仕様: [verify-implementation-spec.md](./verify-implementation-spec.md)

---

## 2. 出典確認（verify-source-attribution-spec）

会話で言及される文章（Backlog コメント・チャット履歴・ドキュメント引用）の出典を断定する前に、必ず確認する。記憶で出典を捏造しない。

**確認が必要な場面**:
- 「この文章は私が書いた／お客さんが書いた」等の出典帰属
- Backlog コメントを引用・解釈する場面 → `mcp__backlog__get_issue_comments` で再取得して照合
- 確認できない場合は `[出典不明]` を付ける

詳細仕様: [verify-source-attribution-spec.md](./verify-source-attribution-spec.md)

---

## 3. スコープ管理（answer-scope-spec）

質問に直接答える1行を最初に書く。派生事項は分離して承認なしに実施しない。

**禁止パターン**:
- 質問されていないコード書き換え・リファクタを無断実施
- 「ついでに〜も直しておきました」系の追加作業
- 関係ない箇所の情報を本文に混入

**派生事項の扱い**:
```
## 派生事項（質問外）
〜についても変更が必要と思われます。対応しますか？
```

詳細仕様: [answer-scope-spec.md](./answer-scope-spec.md)

## 4. 確証なし時のマーカー（uncertainty-marker-spec）

実装・資料確認なしに断定できない場合、以下のマーカーを付ける。根拠なしの断定は禁止。

| マーカー | 使いどころ |
|---|---|
| `[推定]` | 記憶・推論で回答しているが、実装・資料確認が取れていない |
| `[要確認]` | 人間（ユーザーまたはチーム）の確認が必要。Claude が答えを持っていない |
| `[出典不明]` | 出典を確認できなかった発言・引用 |

**鉄則**: 間違った自信ある回答 > 正直な「わかりません」ではない。**逆**。

詳細仕様: [uncertainty-marker-spec.md](./uncertainty-marker-spec.md)（新設予定）

## 5. docs がない場合の振る舞い

sf-context-loader から「該当コンテキストなし」が返った場合 / docs/ が未整備の場合:

- プロジェクト固有事項（命名・業務ルール・組織設定・対応経緯）を **記憶で答えない**
- 「docs に記載がありません。〜を確認してください」と明示する
- 一般的な Salesforce の知識で代替できる部分は `[推定: SF 一般]` を付けて回答可
- 推測で断定しない。「たぶん〜」「おそらく〜」で断定する応答は禁止
