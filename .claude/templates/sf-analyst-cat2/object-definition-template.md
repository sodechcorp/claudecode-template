# カテゴリ2 オブジェクト定義書 テンプレート

---

## オブジェクト定義書の必須構成

各オブジェクトに対して `docs/catalog/{standard|custom}/<オブジェクト名>.md` を以下の構成で生成する。

```markdown
# {オブジェクト名}（{API名}）

## 基本情報
| 項目 | 値 |
|---|---|
| オブジェクト種別 | カスタム / 標準 |
| 用途 | （業務的な用途を具体的に記述） |
| レコード件数 | {件数} |
| 関連UC | UC-XX: {UC名}、UC-XX: {UC名} |
| 関連FR要件 | FR-XXX、FR-XXX |

## リレーション
（Mermaid ER図を含める。このオブジェクトを中心に、親・子・参照先を全て図示。
  業務リレーション（カスタム間・業務標準オブジェクトへの参照）が無く、Owner/CreatedBy/LastModifiedBy 等の標準監査項目のみの場合は、空の erDiagram ブロックを出力せず `（業務リレーションなし。標準監査項目のみ）` と記載する）

## 全項目一覧
（標準項目・カスタム項目を分けて全量記述。
  型・必須/任意・ユニーク・デフォルト値・用途を列として持つ）

## ピックリスト値
（ピックリスト項目ごとに全値を列挙）

## 入力規則
（名前・条件（数式全文）・エラーメッセージ・有効/無効）

## 自動化（このオブジェクトに関連するApex/Flow）
（どのApexクラス・Flow・トリガーがこのオブジェクトを操作するか）

## 権限マトリクス
（ObjectPermissions を取得していない場合は `**[要確認]**` を使わず、以下の定型注記のみ記載する）

プロファイル/権限セット別 CRUD・FLS は組織のプロファイル/権限セットメタデータで管理されるため本書では収録対象外。権限・FLS を変更した場合は本セクションに追記する。

## 所見・注意点
```

## cross-reference の記載（重要）

「このオブジェクトがどのUCで使われるか」「どのApex/FlowがSOQL/DMLで操作するか」を必ず記載する。情報がない場合は `**[要確認]**` を入れる。

## 備考列マーカー規約

「全項目一覧」テーブルの備考列に使用するマーカー。sf-memory-quality.md のマーカー文化と統一する。

| マーカー | 意味 | 付与タイミング |
|---|---|---|
| `**[本番デプロイ待ち]**` | Sandbox/dev 組織に存在するが本番組織に未反映の項目 | Phase 1.5 で本番突合モードを選択し、Phase 2.6 の差分計算で検出された場合 |
| `**[推定]**` | メタデータに明記がなく、用途・意味を推測した箇所 | describe 結果から業務的意味を補間した場合（**`object-meta.xml` の `<description>` や `masterLabel` をそのまま転記する場合は事実扱いのため `[推定]` 不要**） |
| `**[要確認]**` | 情報が取得できなかった・意図が不明な箇所 | describe 失敗・設計者確認が必要な場合 |

**記載例**（備考列が空の場合）:
```
| IsCriminal__c | 反社チェックフラグ | チェックボックス | | **[本番デプロイ待ち]** |
```

**記載例**（備考列に既存の説明がある場合）:
```
| Status__c | ステータス | テキスト | ○ | **[本番デプロイ待ち]** GF-341 対応項目 |
```

> テーブル列構成（API 名 / 表示ラベル / 型 / 必須 / 備考）は変更しない。マーカーは備考列の先頭に置く。

## セル内テキスト整形規約（数式・Help テキスト・regex・長文）

「全項目一覧」「入力規則」「ピックリスト値」等のテーブルセルに数式・Help テキスト・regex を書く際は以下を厳守する（docs を再利用・コピーしても壊れないように）。

| 対象 | ルール |
|---|---|
| 改行を含む値（Help テキスト・複数行数式） | セル内の生改行は禁止。改行は `<br>` に変換して 1 セル 1 行に収める（生改行はテーブルを崩す） |
| Salesforce regex / 数式の `\b` `\n` `\r` `\d` 等 | `sf ... --json` 出力では `\b` が `\\b` のように二重エスケープされる。**md には Salesforce 上の実値（`\b` `\n`）を書く**。JSON 由来の余分な `\` を引き継がない |
| 数式・条件式の全文 | `...` で途中省略しない（`IF( AND(ISPICKVAL(Ag...` のような切断は禁止）。全文を記載する |
| 512 文字以上の長い数式・本文 | テーブルセルに直書きせず `<details><summary>数式全文</summary>` … `</details>` の折りたたみブロック、または項目直下の別ブロック（コードフェンス）に切り出す |

> これらは個別オブジェクト定義書（`custom/`・`standard/` 配下）に適用する。下流 `generate_basic_doc.py` は個別定義書を読まないため整形の自由度は高いが、人間・LLM の再利用時に壊れないことを最優先する。

## `_index.md` 列規約（下流パーサー連携）

`_index.md` のオブジェクト一覧テーブルは以下のキーワードをヘッダ行に含めること（列順は問わない）:

| キーワード | 必須 | 用途 |
|---|---|---|
| `API名` | 必須 | 下流 `generate_basic_doc.py` が識別キーとして読む |
| `ラベル` | 必須 | 日本語表示名。ER図のノード表示に使用 |
| `種別` | 推奨 | `カスタム` / `標準` |

---

## Mermaid ERD 記法の規約（下流パーサー連携・厳守）

`generate_basic_doc.py` が正規表現で関係線を抽出するため以下を守ること。

**関係線の構文**:
```
ParentApiName ||--o{ ChildApiName : "fieldLabel"
```

- 実線 `--` に統一（点線 `..` も受理されるが原則実線）
- 右端カーディナリティ: `o{` / `|{` / `||` / `o|` のいずれか
- ラベル部分（`:` の右）は **FK項目の API名 または日本語ラベルを必ずダブルクォートで囲む**（空文字列は不可）
- **標準オブジェクトも含める**: Account / Contact / Opportunity / Case / Lead 等、カスタムから参照されているものは必ずノードに含める
- **空ブロック禁止**: 描画すべき業務リレーションが無い（参照が Owner / CreatedBy / LastModifiedBy 等の標準監査項目のみ）場合は、`erDiagram` だけの空ブロックを出力しない。`（業務リレーションなし。標準監査項目のみ）` の 1 行に置換する
- **親子方向は describe で確定する（推測しない）**: Lookup/Master-Detail 項目を**持つ側**が子（`||--o{` の右）、その項目の `ReferenceTo`（参照先）が親（左）。Phase 2 で取得した `FieldDefinition`（`ReferenceTo` / `RelationshipName`）の実測で向きを決め、オブジェクト名や業務上の上下関係から推測しない。同一リレーションを複数の定義書に描く場合も全ファイルで親子方向を一致させる（個別 ER 図と `_data-model.md` の矢印が食い違わないように）

**悪い例**（関係線がマッチしない）:
```
Account -- ContractApplication__c
Account o-- Contract
```

**良い例**:
```
Account ||--o{ ContractApplication__c : "AccountId__c"
Opportunity ||--o{ ContractApplication__c : "OpportunityId__c"
```
