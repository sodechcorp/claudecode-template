# sf-memory 品質原則・マーカー規約（全カテゴリ共通）

sf-analyst-cat1〜cat6、および sf-org-analyst が共通して守る原則。cat8 は外部 SF 公式仕様を扱うため独立した品質原則を持つ（sf-analyst-cat8.md 参照）。

## 品質原則

1. **網羅的に読む**: 指定資料・ソースコードは配下を再帰的に**全て**読む。サンプリングや抜粋禁止。大きいファイルは分割読みで**最後まで**目を通す。
2. **事実と推定を分ける**: メタデータ・資料・コードに明記されている事項は事実として記述。補間・推測した箇所は不確実マーカー（下記参照）を付ける。空欄を勝手に埋めない。
3. **手動追記を消さない**: 差分更新モードでは既存の手動記入・設計コメント・要件番号・判断根拠を絶対に保持する。

## マーカー規約

全カテゴリで使用可・全カテゴリで検出対象とするマーカー一覧。各エージェントの個別マーカー定義はこの規約に統一し、追加定義は不要。

| マーカー | 意味 |
|---|---|
| `**[要確認]**` | 確認が必要（SF 組織・設計書・ユーザーへの問い合わせが必要） |
| `**[推定]**` | 補間・推測した内容（事実として断定できない） |
| `**[資料未確認]**` | 指定資料に該当記載なし（存在するはずだが見当たらない） |
| `**[組織未調査]**` | メタデータ・ソースコードで未確認 |
| `**[未ヒアリング]**` | ユーザー・ステークホルダーへの確認が取れていない |
| `**[出典不明]**` | 記載の出所が辿れない |
| `**[未実装]**` | ソースが欠落・実装が途中の状態 |
| `**[要確認: <理由>]**` | コロン後に理由を付与する複合形（例: `**[要確認: 他ファイルと矛盾あり]**`） |
| `**[推定: <理由>]**` | 推定の根拠・分布情報を併記する複合形（例: `**[推定: 分布=Customer:65%/Partner:25%/Other:10%]**`、`**[推定: UC-01 / UC-03 のいずれか]**`）。確認情報を増やすことで将来の人間確認を容易にする |

**sf-org-analyst Phase 4 / Phase 6 の検出対象**: 上記 7 種すべてを Grep 対象とする（`\[要確認\|推定\|資料未確認\|組織未調査\|未ヒアリング\|出典不明\|未実装\]`）。

## 技術識別子禁止の原則（業務本文セクション向け）

業務担当者向けの自然文（説明文・タイトル・トリガー・備考等）には、
**プロジェクトに初めて参加した人間が読んで意味が分かる言葉のみ** を書く。

これは「英語禁止」ではない。業務担当者が日常使うサービス名・製品名は **業務固有名詞** として許容する。

### ❌ 禁止する技術識別子

- API 名: `DailyReport__c` 等 `*__c` 形式
- Apex クラス名: `SkyBookingController`・`sdch_Trigger_*`・`sdch_Batch_*` 等
- Aura/LWC/Flow API 名: `SkyFactoryCalendar`・`sdch_Flow_*` 等
- 併記接尾辞: `（Aura: XYZ）`・`（Apex: XYZ）`・`（LWC: XYZ）`・`（Flow: XYZ）`

### ✅ 許容する語

- 業務固有名詞: `Pardot` / `GMOサイン` / `OPROARTS` / `Reserve サイト` / `LINE` / `SMS` / `Chatter` 等（業務担当者が日常使う名前）
- SF 一般用語: `Experience サイト` / `承認プロセス` / `権限セット` / `カスタム通知` 等
- 業務略号: `URL` / `PDF` / `CSV` / `Web` / `Excel` 等（title 等の **一部として** 含まれる場合は NG — 例: `友達追加URL` は URL が操作名に紛れている）
- 業務 ID: `UC-XX` / `FR-XXX` / `CMP-XXX`

### 適用するフィールド

本原則を適用する具体的フィールドは各カテゴリのエージェント定義（Phase 0.5 ブロック）に書く。
書かれていない field には適用しない（過剰検出を防ぐ）。
以下は **適用外**:
- `docs/catalog/*.md` の項目一覧テーブル（API 名カラム）
- `docs/knowledge/sf-standard.md`（cat8 成果物・SF 公式仕様）
- `docs/knowledge/case-index.md` の「関連用語」カラム（cat6 が意図的に API 名を記録）

### 差分更新時の動作

差分更新モードで本ルール違反を検出した場合:
1. `org-profile.md` 用語集または既存表記から業務語を引いて置換する
2. 引けない場合は `**[要確認: 業務語へ変換要]**` マーカーを付与する
3. **規約適合は手動追記より優先する**（cat1 と同方針）

## [推定] マーカー自動解消基準

cat2 Phase 2.5 / sf-org-analyst Phase 4-A で参照する共通判定表。判定は `docs/.sf/_picklist_samples.json`（cat1 Phase 1-6 で生成）のキャッシュのみを参照する。追加クエリは行わない。

| 条件 | 判定 | 出力形式 |
|---|---|---|
| 単一値が ≥70% を占める | 主用途断定可能 | `用途: <その値ラベル>（{X}%）` — `**[推定]**` 付けない |
| 有効値が 3 以下 | 値が少なく全値列挙可能 | 値全列挙で事実記述 — `**[推定]**` 付けない |
| 全レコード空欄率 ≥90% | ほぼ未使用 | `ほぼ未使用（空欄率 {X}%）` — `**[推定]**` 付けない |
| 上位 3 値の合計 ≥95% | 主用途 Top3 に集中 | Top3 列挙で事実記述 — `**[推定]**` 付けない |
| 上記条件に合致しないが分布データあり | 用途断定不可 | `**[推定: 分布={値A}:{X}%/{値B}:{Y}%/...]**` で残置（分布併記リッチ化） |
| キャッシュにない | データなし | `**[推定]**`（従来通り・変更なし） |

**70% 閾値の理由**: 80% は厳しすぎ（Account.Industry 等の現実分布で大半が残置になる）。50% は弱い推定の事実化リスクが高い。プロジェクト固有の許容度が異なる場合はこのファイルのコメントに上書き値を記載すること。

**適用スコープ**: ピックリスト項目の「用途」欄のみ。テキスト型・数値型・リレーション型の推定には適用しない。

## 件数・数値の一貫性原則（差分更新・横断補完共通）

cat1/2/3 の差分更新フェーズおよび sf-org-analyst の横断補完フェーズが共通して守る原則。

### 正本（source of truth）と同期先

| 数値の種類 | 正本 | 同期先 |
|---|---|---|
| オブジェクト別レコード件数 | `data/data-statistics.md`（無ければ `catalog/{obj}.md` 基本情報テーブル） | org-profile 構成サマリ・catalog/_index.md・catalog/{obj}.md 用途/所見欄・usecases.md |
| 項目数 | `catalog/{obj}.md` 項目一覧の実カウント | org-profile・requirements・catalog/_index.md |
| ユーザー総数・プロファイル別人数 | `org-profile.md`（cat1 Phase 1-2 実測） | requirements・その他本文 |
| レポート/ダッシュボード件数 | `data/reports-dashboards.md` 本文テーブル | data/_index.md・org-profile |
| オブジェクト総数（カスタム/標準別） | `catalog/custom/`・`catalog/standard/` のファイル数（分類定義は cat2 Phase 1「オブジェクト分類定義」） | `catalog/_index.md` 種別集計・`catalog/_data-model.md` 全体統計・`org-profile.md` 構成サマリ |

**優先順位**: 専用集計ファイル > 個別定義書の集計テーブル > プロズ/用途欄/所見欄。同一ファイル内は 集計テーブル > 本文。

### 適用ルール

1. **差分更新時（cat1/2/3）**: 正本値を書いたら**同一ファイル内**の旧件数を Grep し正本値で上書きする（数値のみ差し替え・手動コメント保持）。
2. **横断補完時（sf-org-analyst）**: 正本↔同期先を突合し正本値に統一する。
3. **版管理**: 旧値は本文から削除し `<!-- 旧値: XXX（YYYY-MM-DD 時点）-->` か改版履歴へ。新旧 2 値を本文に併存させない。
4. **判定不能時**（正本未生成・正本候補が複数で不一致）: 両所に `**[要確認: 他ファイルと件数矛盾]**` を付与し残置する（推測で揃えない）。

**検出パターン**: `[0-9][0-9,]*\s*件` / `[0-9][0-9,]*\s*名` / `[0-9][0-9,]*\s*項目` / `[0-9][0-9,]*\s*個`
